Getting Started With the REST API

This guide describes how to build a Taxonomy Hub example application using the Carbon LDP REST API. The REST API provides a way for applications to interact with the Carbon platform using HTTP verbs (GET, POST, PUT, DELETE, etc.) and resources identified by URIs. Although developers may prefer to use the App Dev UI and the JavaScript SDK, all functions of the platform are available through the REST API in a way that conforms to the W3C Linked Data Platform 1.0 specification.

Pre-release Documentation

Although Carbon LDP is not yet released, we're publishing documentation as it is prepared so that you can preview what's in store.

Note: Some URI's featured in this documentation may use the "{{host}}" domain, which is inoperable.

About the application

A taxonomy is a controlled vocabulary of terms, organized in a hierarchy, that can be used for classifying and organizing information. Taxonomies are commonly used in web applications. Their terms are used for tagging content and as facets to help users filter and find. In this guide, we'll use the REST API to create a simple, but useful taxonomy service. Once done, you'll understand some of the the most important functions of the REST API as well as fundamental Linked Data Platform concepts. You'll also have a useful application that you can use to create and manage a single 'canonical' source of terms such as a corporate enterprise taxonomy. Other applications will be able to consume and use the service using the very same API.

In this guide, we'll use the following steps to complete the application:

  1. Create an account on carbonldp.com
  2. Create an application
  3. Model the application (containers, resources, vocabularies)
  4. Create application resources
  5. Create application roles
  6. Assign permissions on your application structure to the application roles
  7. Create one or more application users
  8. Assign users to roles
  9. Optional: Create a front-end user-interface

Since this guide focuses on the REST API, you'll need a tool that can help you create and send HTTP requests. We recommend Postman. It's a great tool, it's free, and it's the one we'll be featuring in this guide. But feel free to use the tool you're most comfortable with.

Create an account on carbonldp.com

To interact with Carbon through the REST API, we'll need an account on carbonldp.com. Even this can be done through the REST API, so let's start by issuing a POST to create a new Carbon account. The resource that comprises a Carbon account is known as an Agent, a platform agent to be precise. Later, you will learn about  application agents, which represent accounts specifically for users of the applications you build.

Create the request

Create the following HTTP request to create a platform account.

POST https://{{host}}/platform/agents/

HTTP Header Value Required?
Content-Type text/turtle Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Slug jane-doe No

Review the request

Before issuing this request, you may want to examine its parts. To get more information about each part in the request, click to expand an item of interest below.

Content-Type

Since the request method is POST, the Content-Type HTTP header tells the platform what kind of content to expect in the body of the request. We use turtle (Terse RDF Triple Language) in our documentation because it's brief and clean. Once you understand the turtle syntax, you'll find that it's very easy to read and write. In all, Carbon supports the following content types:

  • Turtle: text/turtle
  • JSON-LD: application/ld+json
  • RDF XML: application/rdf+xml
Prefer

The Prefer header uses a link relation to specify the interaction model of the target resource, which is the resource we're posting to - the /platform/agents/ container. By using this header, we're specifying the desired behavior of the resource. Because we're adding a member resource to the target container, we need it to behave like a container as opposed to an RDFSource or a NonRDFSource. In all, Carbon supports the following interaction model values for the Prefer header:

  • http://www.w3.org/ns/ldp#Container; rel=interaction-model
  • http://www.w3.org/ns/ldp#RDFSource; rel=interaction-model
  • http://www.w3.org/ns/ldp#NonRDFSource; rel=interaction-model
Slug

The Slug header is intended to give the server a hint as to how to mint a new URI for the resource being created. If a slug is not provided, Carbon will generate a random number for the URI of the agent. In this case, however, we're specifying a preference to use john-doe. Therefore, if it is available for use, the server will mint the following URI:

https://{{host}}/platform/agents/jane-doe/

Body

The body of the request is an RDF graph of triples written in Turtle, which we told the server we'd be using in the Content-Type header.

Using Turtle's @prefix directive we declare short prefix names so that we don't have to repeat long URIs in the triples that follow.

Following is a brief description of each of the required triples that come after the namespace declarations:

 

Subject Predicate Object 
<>
 
a
 
cs:Agent

The subject specifies an empty value relative to the base URI. In turtle, URIs are written enclosed in '<' and '>' and may be absolute RDF URI References or relative to the current base URI.
The token 'a' in the predicate position always specifies the RDF type of the resource, which is a short way of writing:

http://www.w3.org/1999/02/22-rdf-syntax-ns#type

In the object position, we specify the RDF type as being an Agent from the Carbon LDP security vocabulary.

 
<>
 
cs:name
 
"Jane Doe"
The name is a literal string that should specify a common name for the user; we recommend using the first and last name. 
<>
 
vcard:email
 
"jane.doe@example.org"
The email is a literal string that should be a valid contact email address for the user. This will also serve as the login user ID. 
<>
 
cs:password
 
"MyStr0ngP@ssw0rd#"
The password is a literal string that should be the desired password.

Issue the request to create an account

Issue the POST request.

A successful request will result in HTTP status code 201 Created. The ETag header in the response will confirm the time of creation and the Location header will provide the server minted URI for the new Agent resource. For example:

HTTP Header Value
ETag W/"2015-11-09T19:18:30.863Z"
Location https://{{host}}/platform/agents/jane-doe/

Examine the new Agent resource

Now, let's take a look at the newly created resource by issuing a GET request to the URI that was minted for the resource. 

Create the following HTTP request to get the platform Agent.

GET https://{{host}}/platform/agents/jane-doe/

You will need to authorize the request using the username and password of the that was given in the body of the former POST request.

Username john.doe5@example.org
Password MyStr0ngP@ssw0rd#

If you paste the URI in the address bar of your web browser and hit enter, for example, you should be prompted for the username and password. If you issue the request with Postman or another tool, you should add Basic Auth to the request.

Let's also add an Accept header to change the format of the response to JSON LD.

HTTP Header Value Required?
Authorization Basic ... Yes
Accept application/ld+json No (default is text/turtle)

200 OK

Create an application

With your newly created platform account, you can now create an Application.

To create an application, issue the following request. Be sure to use your own unique name for the Slug or omit the Slug. You should also provide a unique name for the application in the body of the request.

GET https://{{host}}/platform/apps/

HTTP Header Value Required?
Authorization Basic ... Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Content-Type text/turtle Yes
Slug taxonomy-hub No

Body (required)

Issue the POST request.

A successful request will result in HTTP status code 201 Created. It will also respond with the server-minted URI for the Application resource, which in the case of our example is:

https://{{host}}/platform/apps/taxonomy-hub/

Examine the application root container

You can always verify a newly created resource by issuing a GET request to its URI, but it's important to keep in mind that the URI created for the Application only references the metadata resource for your application. When a new Application is created, Carbon also creates an access point at the following URI:

https://{{host}}/apps/<app-slug>/

So, in the case of our example, the application's access point or root URI would be:

https://{{host}}/apps/taxonomy-hub/

The difference between this URI and the URI of the Application metadata resource is the absence of the /platform segment after the host name.

Issue the following request to examine the root container that Carbon created for you.

GET https://{{host}}/apps/taxonomy-hub/

HTTP Header Value Required?
Authorization Basic ... Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes

Notice this time that since we did not specify an Accept header of type application/ld+json, so the response we'll get back will be in the default Turtle syntax. While JSON-LD is often easiest to process with JavaScript, you will typically find Turtle to be a little more compact and easier to read.

Example root container for an app

Model the application (vocabulary, resources, containers)

Now that we have an application root created, we can begin to create RDF documents in the application. We could start posting arbitrary RDF documents right away, but it's a good idea to have a general model for the application first. What entities will the resources in our application represent? What data attributes (or triples) will they have. Are there existing RDF vocabularies we can leverage for those attributes? What kinds of relationships (links) will we make in our data or with data in other sources? How will we use container behavior to create, modify, and enumerate member documents? Answering these questions up-front will help provide a logical and structured way to build the application. 

One way to think about this is to think about the most common use cases for the data and the following six areas of consideration:

  1. Classes/Entities - The taxonomy terms are the main item of interest or entity we'll be working with - the main resource we'll be managing in the Carbon app.
  2. Class/Entity Properties - Taxonomy terms are pretty simple; they're just concepts or labels. Perhaps they can be expressed in multiple languages. We'll start simple and let the entities change as more is required.
  3. Class/Entity Relationships -  A taxonomy implies that these terms are organized in a hierarchy, so we'll probably need to think about how to use container behavior to support that structure.
  4. Application Functionality - The data for our application, the taxonomy terms, will typically be metadata used to help describe the subject and nature of information in other systems, so we'll allow other apps to make REST requests to consume the service. 
  5. Security - We'll need varying degrees of access to the data in our app, so we'll need to come up with the appropriate roles for controlling access.
  6. User Interface - It's also likely that we'll want a user interface that makes it easy for permitted individuals to visualize and manage the taxonomy. Such a UI will likely employ the use of a tree control. 

Re-using an established linked data vocabulary called SKOS

It's always a best-practice to re-use established linked data vocabularies instead of (re-)inventing duplicates and as it turns out, there's already a good vocabulary that seems to fit the needs of our application. It's called SKOS (Simple Knowledge Organization System). According to the SKOS Primer, it " provides a model for expressing the basic structure and content of concept schemes such as thesauri, classification schemes, subject heading lists, taxonomies, folksonomies, and other similar types of controlled vocabulary." Sounds right up our alley, doesn't it? When we investigate SKOS, we not only see that it will suit the needs of our app, but that it provides a lot of clues about how we should model everything.

Mini SKOS Simple Knowledge Organization System primer

skos:Concept

SKOS introduces the class skos:Concept, which allows implementors to assert that a given resource is a concept. This is done in two steps:

  1. by creating (or reusing) a Uniform Resource Identifier (URI) to uniquely identify the concept.
  2. by asserting in RDF, using the property rdf:type, that the resource identified by this URI is of type skos:Concept.

So, there you can see, we already have the class that can represent a taxonomy term. This means that when we create resources that represent taxonomy terms, we simply need to assert that it's an rdf:type skos:Concept. For example:

<Carbon-document-URI> a skos:Concept.

(Remember, 'a' in turtle is shorthand for rdf:type).

skos:prefLabel

A SKOS Concept can have a preferred lexical label, which will be useful in representing it in a user interface. We'll be able to define a label for a concept as shown below.

We can also support locale-specific labels.

There is also a skos:altLabel and a skos:hiddenLabel, which can be used in the future, but we'll keep it simple for now and just stick to the skos:prefLabel.

skos:broader and skos:narrower (semantic relationships)

The skos:broader and skos:narrower properties give us what we need to express hierarchical links between our concepts.

To assert that one concept is broader in meaning (i.e. more general) than another, the skos:broader property is used. The skos:narrower property is used to assert the inverse, namely when one concept is narrower in meaning (i.e. more specific) than another. For example:

In SKOS, there is also the notion of a skos:related link, which is used to assert an associative relationship. But again, we can add such things in the future. For now, we'll just stick to the parent/child relationship afforded by skos:broader and skos:narrower.

skos:inScheme

The skos:inScheme property is another we'll in concept documents. This property will allow us to associate a concept to one or more taxonomies. In our app, taxonomies can be represented by the skos:ConceptScheme, which is described next.

skos:ConceptScheme

The skos:ConceptScheme class will represent a taxonomy that concepts belong to. We'll use two properties in with this class:dct:title and skos:hasTopConcept.

dct:title

The dct:title property is actually from another popular vocabulary - the Terms vocabulary of the Dublin Core Metadata Initiative. We'll use it to add a title to the taxonomies we create.

skos:hasTopConcept

The skos:hasTopConcept property will provide an efficient access to the entry points of our broader/narrower concept hierarchies. This property will allow our app to link a concept scheme (a taxonomy in our case) to the concepts it contains. The top concepts will represent the root concepts in the taxonomies we create.

Consider the role of LDP containers

It may seem intuitive to model the container behavior for our application in such that each concept is a container to its narrower concepts. Or, in other words, so that narrower concepts are represented as membership triples on their broader concepts.

Let's imagine the following simple taxonomy, for example:

  • Topics (ConceptScheme)
    • Design (Top Concept 1)
      • Information Architecture (Concept 1.1)
    • Technology (Concept 2)
      • HTML5 (Concept 2.1)
      • JavaScript (Concept 2.2)

The intuitive container behavior to model might be as follows:

  • taxonomy-hub/topics/
    • design/
      • information-architecture/
    • technology/
      • html5/
      • javascript/

Though Carbon enforces no restriction on doing this, there are a few reasons why we might not want to.

First, as the depth of your taxonomy increases, so too does the length of your concept URIs. We should keep in mind that the fully qualified URI for each concept will also include the host and the base URI to the app. In this model, just one level deeper from the top concept could produce the following URI:

http://dev.carbonlp.com/platform/apps/taxonomy-hub/topics/development/javascript

You can see how the addition of just a few more levels could lengthen the URI significantly.

Another possible problem is that it creates a tight coupling between the URI for each concept and the location of the concept in a hierarchy. When you move categories around, you'd have to revise the URIs for several other concepts to keep the implication in tact. Furthermore, if you ever wanted to share concepts between branches of a taxonomy, or across multiple taxonomies, the implication of the URI would only be logical in one of the cases. 

Finally, there is a more elegant approach for leveraging container behavior that can reduce the work required to manage the taxonomies.

  • taxonomy-hub/schemes/
    • topics/
      • top-concepts/
      • design/
      • information-architecture/
      • technology/
      • html5/
      • javascript/

We've flattened things out a bit so that all concepts are direct children of the topics/ container. We've also added a new container called top-concepts to help us identify and manage which, of all the concepts, are those that should be root concepts in the scheme. 

As we work with this alternative approach, you'll begin to see why it's an improvement.

Create the schemes/ container

Let's start by creating the schemes/ container. Since the application will always have only one schemes/ container, this will be a one-time affair.

Create the following request.

POST https://{{host}}/apps/taxonomy-hub/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type text/turtle Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Slug schemes No

Body (required)

We're specifying a Basic Container so that membership will automatically be recorded on the container when new schemes are posted, which will provide an easy way to list available schemes.

Issue the request. A successful response should provide the HTTP status code 201 Created.

Create example Concept Scheme

Now that we have the schemes/ container, we can create one or more Concept Scheme containers within it. Since this is an action that would likely be repeated by end-users, it would make sense to incorporate the capability into a user interface. For now, however, let's be clear on what is required for the underlying HTTP request.

POST https://{{host}}/apps/taxonomy-hub/schemes/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type text/turtle Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Slug topics No

Body (required)

Take note of the following triple on the container:

<> ldp:memberOfRelation skos:inScheme

This configures the container to automatically add a link in each member that is added to the container. Whenever we POST a new concept into this scheme, Carbon will automatically add the following triple to the concept document:

<concept-uri> skos:inScheme <https://{{host}}/apps/taxonomy-hub/schemes/>

Create and issue the request. A successful request should result in HTT status 201 Created.

This illustrates an interesting use of BasicContainer behavior. In the next step, we'll see another interesting use of container behavior with a DirectContainer.

Create the top-concepts/ Direct Container

As stated earlier, the top-concepts/ container will give us an easy way to list the root concepts in a taxonomy. Our application should therefore create a top-concepts/ container in every scheme/ created. You'd likely want to automate the creation of this container in the function that creates the scheme/ container, but for now, we'll do it by hand to complete the exercise.

This time, we'll make use of a Direct Container. A Direct Container creates an access point that helps you manage membership triples in another container.

Create the following request.

POST https://{{host}}/apps/taxonomy-hub/schemes/topics/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type text/turtle Yes
Prefer http://www.w3.org/ns/ldp#RDFSource;rel=interaction-model Yes
Slug top-concepts No

Body (required)

Notice, this time, we use the Prefer header to set the interaction model to RDF Source instead of Container. This keeps the topics/ container that we're posting to from recognizing the top-concepts/ container as a member. That is to say, it keeps it from recording a membership triple for the top-concepts/ container. The outcome we want is for all concepts to be listed as members of the topics/ container, but not the top-concepts/ container.

The ldp:membershipResource predicate of this Direct Container specifies an entirely different resource where membership triples should be written. Whenever we post to this container (the top-concepts/ container), a membership triple will be written to the specified resource, which is the topics/ container. In other words, if we later put a Concept to the top-concepts/ container, that concept will become a member of the topics/ container. Why would we want to do this? To understand, we have to consider the ldp:hasMemberRelation predicate, which is also used.

Whenever the top-concepts/ container writes a membership triple to the container specified by ldp:membershipResource, it will use the predicate skos:hasTopConcept to represent that membership relationship because of what's defined by ldp:hasMemberRelation. This feature exists so that we can use a different vocabulary other than the standard LDP vocabulary to represent membership relations. When we post to top-concepts/, for example, the following statement will be added to the topics/ container:

<https://{{host}}/apps/taxonomy-hub/schemes/topics/> skos:hasTopConcept <member-URI>

We will not be posting to the top-concepts/ container to create new documents. Instead, we'll post new concepts to the topics/ container to create them and then later put the top concepts into the top-concepts/ container. If you post a new resource to a container, the container creates a membership triple and a containment triple, but if you put an existing resource to a container, the container creates only a membership triple.

Issue the request. A successful response should provide the HTTP status code 201 Created.

Create a root Concept

Now, we're ready to start creating concepts in our Topics taxonomy. This, of course, is a function that the GUI of our app would probably handle, but doing a few manually is important to this learning exercise.

For our first Concept, create the following request.

POST https://{{host}}/apps/taxonomy-hub/schemes/topics/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type text/turtle Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Slug design No

Body (required)

Notice, this time that the ldp:hasMemberRelation is skos:narrower. That means that any concept that is later PUT to this container will force a membership triple to be written on the container using skos:narrower as the predicate.

The ldp:memberOfRelation allows the inverse to be true. Any concept that is added to the membership of this container will get a triple written to it, which defines this concept as its skos:broader concept. When we add the Information Architecture concept as a member, for example, the information-architecture/ resource will receive the following triple. 

<...topics/design/information-architecture/> skos:broader <......topics/design/>

As we can see now, features of a container support what's needed to use the SKOS vocabulary. From any concept, we have a way to look both downwards and upwards in the hierarchy.

Issue the request. A successful response should provide the HTTP status code 201 Created.

Add root Concepts as member of the scheme

Now that the root concept is created, it is a member of its parent container. In other words, its parent container, topics, links to it with the following membership triple:

<> <http://www.w3.org/ns/ldp#member> <https://{{host}}/apps/taxonomy-hub/schemes/topics/design/>

Since the parent container is a BasicContainer, the concept is also contained by it. The parent container expresses this relationship with the following containment triple:

<> <http://www.w3.org/ns/ldp#contains> <https://{{host}}/apps/taxonomy-hub/schemes/topics/design/>

What's needed now is to identify the root concept as a member of the topics scheme with the skos:hasTopConcept relation. This will be done when we PUT the concept to the top-concepts container.

Create the following request.

PUT https://{{host}}/apps/taxonomy-hub/schemes/topics/top-concepts/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type text/turtle Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes

Body (required)

Notice that the Carbon vocabulary is used. The class of the resource is a Carbon AddMemberAction with the specified member to add to the container we're putting it to.

Issue the request. A successful response should provide 200 OK.

Now, if you examine the {{host}}/apps/taxonomy-hub/schemes/topics/ container (GET), you'll see the membership triple link to the design/ concept written as skos:hasTopConcept.
