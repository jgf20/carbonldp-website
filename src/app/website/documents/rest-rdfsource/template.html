RDFSource

Pre-release Documentation

Although Carbon LDP is not yet released, we're publishing documentation as it is prepared so that you can preview what's in store.

You can use the Carbon LDP REST API to create, read, update, and delete RDFSource documents. Here's how...

Create

To create a new RDFSource, POST the new document to the URI of a container.

POST /apps/{app-name}/{container}/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type Either of:
  • text/turtle
  • application/ld+json
  • application/rdf+xml
  • application/trig
 Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Slug {john-doe} No

Body (required)

The RDFSource document is required in the entity body of the request. Following is an example RDFSource document in TRIG syntax. The syntax used for the body must match what you declare in the Content-Type header. 

						@prefix ldp:<http://www.w3.org/ns/ldp#>.
						@prefix ex:<http://www.base22.com/2016/05/ex#>.

						<> {
							<>
								a ldp:RDFSource, ex:Person;
								ex:firstName "John";
								ex:lastName "Doe";
								ex:phoneNumber "(555) 555-5555".
						}

Issue the request. A successful response should provide the HTTP status code 201 Created.

If the optional Slug header is provided, the server will attempt to mint a URI for the document by appending the slug to the path. If the slug is not provided, the server will generate a random long number for the URI.

The document's fully qualified URI will be returned by the Location header in the response. Whether or not the server mints the URI using a slug or random long number, the URI will always end with a trailing forward-slash.

An ETag header will also be returned, which can be used to determine if the document has changed since it was first created or last modified.

Example request to create RDFSource :

						POST /apps/rest-examples/persons/ HTTP/1.1
						Authorization: Basic
						Content-Type: application/trig
						Prefer: http://www.w3.org/ns/ldp#Container; rel=interaction-model
						Slug: john-doe

						@prefix ldp:<http://www.w3.org/ns/ldp#>.
						@prefix ex:<http://www.base22.com/2016/05/ex#>.

						<> {
						    <>
						        a ldp:RDFSource, ex:Person;
						        ex:firstName "John";
						        ex:lastName "Doe";
						        ex:phoneNumber "(555) 555-5555".
						}

Read

To read an RDFSource, issue a GET request to the document's URI.

GET /apps/{app-name}/{container}/{rdfsource}/

HTTP Header Value Required?
Authorization Basic ... Yes
Accept Either of:
  • text/turtle
  • application/ld+json
  • application/rdf+xml
  • application/trig
 No
Prefer http://www.w3.org/ns/ldp#RDFSource; rel=interaction-model Yes

A successful request will result with HTTP status code 200 OK.

Example response from reading an RDFSource :

Following is the response from reading a newly created RDFSource (from the "Create" example above).

						<https://dev.carbonldp.com/apps/rest-examples/persons/john-doe/>
							a <http://www.w3.org/ns/ldp#BasicContainer> ,
							  <http://www.w3.org/ns/ldp#Container> ,
							  <http://www.w3.org/ns/ldp#RDFSource> ,
							  <http://www.base22.com/2016/05/ex#Person> ;
						<http://www.w3.org/ns/ldp#hasMemberRelation> <http://www.w3.org/ns/ldp#member> ;
						<https://carbonldp.com/ns/v1/platform#created> "2016-05-31T23:29:18.624Z"^^<http://www.w3.org/2001/XMLSchema#dateTime> ;
						<https://carbonldp.com/ns/v1/platform#modified> "2016-05-31T23:29:18.624Z"^^<http://www.w3.org/2001/XMLSchema#dateTime> ;
						<https://carbonldp.com/ns/v1/security#accessControlList> <https://dev.carbonldp.com/apps/rest-examples/persons/john-doe/~acl/> ;
						<http://www.base22.com/2016/05/ex#firstName> "John" ;
						<http://www.base22.com/2016/05/ex#lastName> "Doe" ;
						<http://www.base22.com/2016/05/ex#phoneNumber> "(555) 555-5555" ;
						<https://carbonldp.com/ns/v1/platform#defaultInteractionModel> <http://www.w3.org/ns/ldp#RDFSource> .

You may notice that additional properties (triples) are added to an RDFSource when it is created (e.g. created, modified, accessControlList). These are called server managed properties, which Carbon adds automatically when you post new RDFSource documents.

You may also notice that, in addition to the type RDFSource, the resource is now also a BasicContainer. This is a convenient feature of Carbon LDP. When an RDFSource is created, it is automatically treated also as container. Therefore, you can read an RDFSource as an RDFSource document and a container depending on the interaction model you provide in the Prefer header of the GET request. If you're interested in the properties of the resource (as an entity), you should prefer the RDFSource interaction model when you issue the GET request. If you're interested in the membership of the resource as a container, you should prefer the container interaction model when you issue the GET request.

Update

To update an existing RDFSource, issue a PUT request to the URI of the document to replace it with the document being submitted.

PUT /apps/{app-name}/{container}/{rdfsource}/

HTTP Header Value Required?
Authorization Basic ... Yes
Content-Type Either of:
  • text/turtle
  • application/ld+json
  • application/rdf+xml
  • application/trig
 Yes
Prefer http://www.w3.org/ns/ldp#RDFSource; rel=interaction-model Yes
