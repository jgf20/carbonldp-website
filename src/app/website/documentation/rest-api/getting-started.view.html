Getting Started with the REST API

This guide describes how to build an example application using the Carbon LDP REST API. The REST API provides a way for applications to interact with the Carbon platform using HTTP verbs (GET, POST, PUT, DELETE, etc.) and resources identified by URIs.
The REST API is considered to be a lower-level API than the JavaScript SDK. Doing things directly with the REST API is generally more tedious than with the JavaScript SDK. However, understanding the REST API provides a lot of insight into how the platform works and can be advantageous for developers who seek to advance their understanding of the platform.

About the example application

In this guide, we'll use the REST API to create a simple, but useful taxonomy service. A taxonomy is a controlled vocabulary of terms, organized in a hierarchy, that can be used for classifying and organizing information. Taxonomies are commonly used in web applications. Their terms are used for tagging content and as facets to help users filter and find. In this exercise, you'll learn some of the the most important functions of the REST API as well as fundamental Linked Data Platform concepts. You'll also have a useful application that you can use to create and manage a single 'canonical' source of terms such as a corporate enterprise taxonomy. Other applications will be able to consume and use the taxonomy service using the very same API.

In this guide, we'll use the following steps to complete the application:

  1. Create an account on carbonldp.com
  2. Create an application
  3. Model the application (vocabulary, resources, containers)
  4. Create application resources
  5. Secure the application

Since this guide focuses on the REST API, you'll need a tool that can help you create and send HTTP requests. We recommend Postman. It's a great tool, it's free, and it's the one we'll be featuring in this guide. But feel free to use the tool you're most comfortable with.

Create an account on carbonldp.com

To interact with Carbon through the REST API, we'll need a Carbon user account. Even this can be done through the REST API, so let's start by issuing a POST to create a new account. The resource that comprises a Carbon account is known as an Agent - a platform agent to be precise. Later, you will learn about  application agents, which represent accounts specifically for users of the applications you build.

Review the request

Before issuing this request, you may want to examine its parts. To get more information about each part in the request, click to expand an item of interest below.

Content-Type

Since the request method is POST, the Content-Type HTTP header tells the platform what kind of content to expect in the body of the request. In all, Carbon supports the following content types:

  • Turtle: "text/turtle";
  • JSON-LD: "application/ld+json";
  • RDF/JSON: "application/rdf+json";
  • RDFa: "application/rdf+xml";
  • TriG: "application/trig";
  • N-Triples: "application/n-triples";
  • N3: "text/n3";
  • TriX: "application/trix";
  • BinaryRDF: "application/x-binary-rdf";
  • N-Quads: "application/n-quads";

While JSON-LD is often easiest to process with JavaScript, you will typically find TriG to be a little more compact and easier to read. So in this guide we use Trig And JSON-LD.

Prefer

The Prefer header uses a link relation to specify the interaction model of the target resource, which is the resource we're posting to - the /platform/agents/ container. By using this header, we're specifying the desired behavior of the resource. Because we're adding a member resource to the target container, we need it to behave like a container as opposed to an RDFSource or a  NonRDFSource . In all, Carbon supports the following interaction model values for the Prefer header:

  • http://www.w3.org/ns/ldp#Container; rel=interaction-model
  • http://www.w3.org/ns/ldp#RDFSource; rel=interaction-model
  • http://www.w3.org/ns/ldp#NonRDFSource; rel=interaction-model
Slug

The Slug header is intended to give the server a hint as to how to mint a new URI for the resource being created. If a slug is not provided, Carbon will generate a random number for the URI of the agent. In this case, however, we're specifying a preference to use jane-doe. Therefore, if it is available for use, the server will mint the following URI:

{{protocolAndHost}}/platform/agents/jane-doe/

Body

The body of the request is an RDF graph of how we want our document, in this case an Agent, to be created.

If you want to know more about document's properties, you can do it by checking the Object Model Documentation

Issue the request to create an account

Issue the POST request.

A successful request will result in HTTP status code 201 Created. The ETag header in the response will help us to provide cache validation and the Location header will provide the server minted URI for the new Agent resource. For example:

HTTP Header Value
ETag W/"652085328"
Location {{protocolAndHost}}/platform/agents/jane-doe/

Examine the new Agent resource

Now, let's take a look at the newly created resource by issuing a GET request to the URI that was minted for the resource.

Create the following HTTP request to get the platform Agent.

GET {{protocolAndHost}}/platform/agents/jane-doe/

Note the trailing slash in the URI. An RDFSource created in Carbon is automatically also a Container (a feature Carbon manages for you), so the server-minted URI always ends with a trailing slash.

You will need to authorize the request using the username and password of the agent that was defined in the body of the former POST request. Both the username and password are case-sensitive; use the same case you used when defining the username and password in the Agent resource.

Username jane.doe5@example.org
Password MyStr0ngP@ssw0rd#

If you paste the URI in the address bar of your web browser and hit enter, for example, you should be prompted for the username and password. If you issue the request with Postman or another tool, you should add Basic Auth to the request.

Create an application

With your newly created platform account, you can now create an Application.

To create an application, issue the following request. Be sure to use your own unique name for the Slug or omit the Slug. You should also provide a unique name for the application in the body of the request.

POST {{protocolAndHost}}/platform/apps/

HTTP Header Value Required?
Authorization Basic ... Yes
Prefer http://www.w3.org/ns/ldp#Container; rel=interaction-model Yes
Content-Type application/trig Yes
Slug taxonomy-hub No

Body (required) 


									@prefix c:			<https://carbonldp.com/ns/v1/platform#>.
									@prefix cs:			<https://carbonldp.com/ns/v1/security#>.

										<>
									        a cs:Application;
									        cs:name "Test App".

Issue the POST request.

A successful request will result in a 201 Http response. this contains a Location header with the server-minted URI for the Application resource, which in the case of our example is:

{{protocolAndHost}}/platform/apps/taxonomy-hub/

Model the application

Now that you have an application root created, you can begin to create RDF documents in the application. Before you start posting arbitrary RDF documents, however, it's a good idea to design a general model for the application. Answering the following questions can help you determine what the model should be.

  • What entities will the resources in our application represent?
  • What data attributes (or triples) will they have?
  • Are there existing RDF vocabularies we can leverage for those attributes?
  • What kinds of relationships (links) will we make in our data or with data in other sources?
  • How will we use container behavior to create, modify, and enumerate member documents?

Answering these questions up-front will help provide a logical and structured way to build your application. One way to think about this is to think about the most common use cases for the data and the following six areas of consideration:

  1. Classes/Entities - The taxonomy terms are the main item of interest or entity we'll be working with - the main resource we'll be managing in the Carbon app.
  2. Class/Entity Properties - Taxonomy terms are pretty simple; they're just concepts or labels. Perhaps they can be expressed in multiple languages.
  3. Class/Entity Relationships - A taxonomy implies that these terms are organized in a hierarchy, so we need to think about how to use container behavior to support that structure.
  4. Application Functionality - The data for our application, the taxonomy terms, will typically be metadata used to help describe the subject and nature of information in other systems, so we'll allow other apps to make REST requests to consume the service.
  5. Security - We'll need varying degrees of access to the data in our app, so we'll need to come up with the appropriate roles for controlling access.
  6. User Interface - It's also likely that we'll want a user interface that makes it easy for permitted individuals to visualize and manage the taxonomy. Such a UI will likely employ the use of a tree control.

Re-using an established linked data vocabulary called SKOS

It's always a best-practice to re-use established linked data vocabularies instead of (re-)inventing duplicates and as it turns out, there's already a good vocabulary that fits the needs of our application. It's called SKOS (Simple Knowledge Organization System). According to the SKOS Primer, it "provides a model for expressing the basic structure and content of concept schemes such as thesauri, classification schemes, subject heading lists, taxonomies, folksonomies, and other similar types of controlled vocabulary." Sounds right up our alley, doesn't it? When we investigate SKOS, we not only see that it will suit the needs of our app, but that it provides a lot of clues about how we should model everything.

Mini SKOS Simple Knowledge Organization System primer

skos:Concept

SKOS introduces the class  skos:Concept , which allows implementors to assert that a given resource is a concept. This is done in two steps:

  1. by creating (or reusing) a Uniform Resource Identifier (URI) to uniquely identify the concept.
  2. by asserting in RDF, using the property rdf:type, that the resource identified by this URI is of type skos:Concept.

So, there you can see, we already have the class that can represent a taxonomy term. This means that when we create resources that represent taxonomy terms, we simply need to assert that it's an rdf:type skos:Concept. skos:prefLabel

Consider the role of LDP containers

It may seem intuitive to model the container behavior for our application in such way that each concept is a container to its narrower concepts. Or, in other words, so that narrower concepts are represented as membership triples on their broader concepts.

Let's imagine the following simple taxonomy, for example:

  • Topics (ConceptScheme)
    • Design (Top Concept 1)
      • Information Architecture (Concept 1.1)
    • Technology (Top Concept 2)
      • HTML5 (Concept 2.1)
      • JavaScript (Concept 2.2)

The intuitive container behavior to model might be as follows:

  • taxonomy-hub/topics/
    • design/
      • information-architecture/
    • technology/
      • html5/
      • javascript/

Though Carbon enforces no restriction on doing this, there are a few reasons why we might not want to.

First, as the depth of your taxonomy increases, so too does the length of your concept URIs. We should keep in mind that the fully qualified URI for each concept will also include the host and the base URI to the app. In this model, just one level deeper from the top concept could produce the following URI:

{{protocolAndHost}}/platform/apps/taxonomy-hub/topics/technology/javascript

You can see how the addition of just a few more levels could lengthen the URI significantly.

Another possible problem is that it creates a tight coupling between the URI for each concept and the location of the concept in a hierarchy. When you move categories around, you'd have to revise the URIs for several other concepts to keep the implication in tact. Furthermore, if you ever wanted to share concepts between branches of a taxonomy, or across multiple taxonomies, the implication of the URI would only be logical in one of the cases.

Finally, there is a more elegant approach for leveraging container behavior that can reduce the work required to manage the taxonomies.

  • taxonomy-hub/schemes/
    • topics/
      • top-concepts/
      • design/
      • information-architecture/
      • technology/
      • html5/
      • javascript/

We've flattened things out a bit so that all concepts are direct children of the topics/ container. We've also added a new container called top-concepts/ to help us manage which concepts are root (top) concepts in the scheme. 

As we work with this alternative approach, you'll begin to see why it's an improvement.

The  ldp:membershipResource predicate of this Direct Container specifies an entirely different resource where membership triples should be written. Whenever we add a member to this container (the top-concepts/ container), a membership triple will be written to the specified resource, which is the topics/ container. In other words, if we later put a Concept to the top-concepts/ container, that concept will become a member of the topics/ container. Why would we want to do this? To understand, we have to consider the  ldp:hasMemberRelation predicate, which is also used.

Whenever the top-concepts/ container writes a membership triple to the container specified by ldp:membershipResource , it will use the predicate skos:hasTopConcept to represent that membership relationship because of what's defined by ldp:hasMemberRelation . This feature exists so that we can use a different vocabulary other than the standard LDP vocabulary to represent membership relations. When we post to top-concepts/ , for example, the following statement will be added to the topics/ container:

<{{protocolAndHost}}/apps/taxonomy-hub/schemes/topics/> skos:hasTopConcept <member-URI>

"skos:hasTopConcept":[ { "@id": "member-URI", } ]

We will not be posting to the top-concepts/ container to create new documents. Instead, we'll post new concepts to the topics/ container to create them and then later put the top concepts into the top-concepts/ container. If you post a new resource to a container, the container creates a membership triple and a containment triple, but if you put an existing resource to a container, the container creates only a membership triple.

Secure the application

Setting up security for a Carbon application involves the following general steps:

  • Create application roles
  • Create application users
  • Assign users to roles
  • Put roles into the role-hierarchy
  • Assign permissions on your application structure to the application roles

The ACL defines a accessControlEntry and an inheritableEntry, each linking to an AccessControlEntry by way of a blank nodes. The entry linked by accessControlEntry defines access to the resource itself. The entry linked by inheritableEntry defines permissions that can be inherited by child resources when the child resources do not override them.

Give the Editor role all permissions and the user role READ permissions.

The best way to modify permissions on a resource using the REST API is to start by getting the ACL for the resource as we've done here. We can then copy and paste that ACL into a text editor, make some manual modifications, and then update the ACL.

Conclusion

This guide described how to build an example application using the Carbon LDP REST API.
