Object Schema

The object schema is used to map prefixes to URIs and to map properties to URIs, forming a vocabulary for their applications. Prefixes are short words or abbreviations that expand to a URI.

Prefixes may be defined as any JSON string different from the JSON-LD keywords.

The object schema describes the properties used in your Documents through the direct use of URIs or through aliases or prefixes that expand to URIs. Allowing properties to be represented by an unambiguous identifier. This helps prevent developers from stepping in each other's vocabularies, and makes data useful in the outside. The URIs used to describe the properties may contain its definitions.

This concept is based on The Context definition of JSON-LD.

Carbon's object schema is made of:

  • Prefixes: simple string that expands to a URI.
  • Properties: JSON object describing a property.

Prefixes

Prefixes may be seen as abbreviations, whose value is a simple string that expands to a URI


	{
	    "skos": "http://www.w3.org/2004/02/skos/core#"
	}

Properties

The value of a property is mapped by a JSON object that may contain an @id, @type, @language, or @container keyword.


	{
	    "title": {
	        "@id": "ex:title",
	        "@type": "xsd:string"
	    }
	}

Types of Object Schemas

There are two types of object schemas:

  • General
  • By type

Both types of object schemas can be defined using the methos extendsObjectSchema() from the Carbon.SDKContext.Class class.

General object schema

This object schema, when defined, can be used by every Carbon Document in your application. As long as the Documents do not have a specific object schema defined.

The next example shows how to define a general object schema


carbon.extendObjectSchema( {
	"acl": "http://www.w3.org/ns/auth/acl#",
	"api": "http://purl.org/linked-data/api/vocab#",
	"c": "http://carbonldp.com/ns/v1/platform#",
	"cs": "http://carbonldp.com/ns/v1/security#",
	"cp": "http://carbonldp.com/ns/v1/patch#",
	"ex": "http://example.org/ns#",
	"foaf": "http://xmlns.com/foaf/0.1/",
	"geonames": "http://www.geonames.org/ontology#",
	"gr": "http://purl.org/goodrelations/v1#",
	"http": "http://www.w3.org/2006/http#",
	"ldp": "http://www.w3.org/ns/ldp#",
	"rss": "http://purl.org/rss/1.0/",
	"sd": "http://www.w3.org/ns/sparql-service-description#",
	"skos": "http://www.w3.org/2004/02/skos/core#",
	"types": "http://rdfs.org/sioc/types#",
	"vcard": "http://www.w3.org/2001/vcard-rdf/3.0#",
	"xsd": "http://www.w3.org/2001/XMLSchema#",
	"title": {
		"@id": "ex:title",
		"@type": "xsd:string"
	},
	"name": {
		"@id": "foaf:name",
		"@type": "xsd:string"
	}
} );

Object schema by type

Object schemas can be defined by types. A Document can be decribes by any number of types. For example, a Document is a Container......


carbon.extendObjectSchema( "http://example.org/ns#BlogPost" , {
	"acl": "http://www.w3.org/ns/auth/acl#",
	"api": "http://purl.org/linked-data/api/vocab#",
	"c": "http://carbonldp.com/ns/v1/platform#",
	"cs": "http://carbonldp.com/ns/v1/security#",
	"cp": "http://carbonldp.com/ns/v1/patch#",
	"ex": "http://example.org/ns#",
	"foaf": "http://xmlns.com/foaf/0.1/",
	"geo": "http://www.w3.org/2003/01/geo/wgs84_pos#",
	"http": "http://www.w3.org/2006/http#",
	"ldp": "http://www.w3.org/ns/ldp#",
	"owl": "http://www.w3.org/2002/07/owl#",
	"rdf": "http://www.w3.org/1999/02/22-rdf-syntax-ns#",
	"sfn": "http://www.w3.org/ns/sparql#",
	"skos": "http://www.w3.org/2004/02/skos/core#",
	"types": "http://rdfs.org/sioc/types#",
	"xsd": "http://www.w3.org/2001/XMLSchema#",
	"title": {
		"@id": "ex:title",
		"@type": "xsd:string"
	},
	"tags": {
		"@id": "ex:tag",
		"@type": "@id",
		"@container": "@set"
	}
} );
