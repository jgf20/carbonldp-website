Getting Started With the JavaScript SDK

Pre-release Documentation

Although Carbon LDP is not yet released, we're publishing documentation as it is prepared so that you can preview what's in store.

This guide introduces you to the use of the Carbon Javascript SDK. In it you will findo your installing guide, and the basic methods needed to start using our platform

Prerequisite Knowledge

To better understand this guide and its code examples, a working knowledge of the following prerequisite topics is required.

  • HTML, CSS, JavaScript: See: Mozilla Developer Network.
  • Promises: A standard-based way of handling asynchronous JavaScript logic. See: Promise object (Mozilla Developer Network).

Knowledge on the following prerequisite topics is highly recommended:

  • TypeScript: Carbon's JavaScript SDK is written in TypeScript, a JavaScript superset language, maintained by Microsoft. See: TypeScript.
  • Package Managers: Carbon recommends JSPM
  • ES2015 Modules: ES2015 modules let you "import" a file from another one so you can modularize your application. See: Modules (Exploring ES6, by Dr. Axel Rauschmayer)

Installing the SDK

Installing the SDK depends on what package manager you are using:

Package manager?

If you are not using a package manager you can clone the source code with the following command:

git clone https://github.com/CarbonLDP/CarbonLDP-JS-SDK.git

It already includes a pre-compiled version of the source code so you can just jump right in.

npm 
npm install carbonldp --save
jspm 

jspm install npm:carbonldp
# We also need to install one of carbonldp's dependencies
jspm install npm:jsonld

After it has been installed the way you'll include it depends on your development environment:

Browser - Global Window Object

The SDK comes with a bundled version that you can import directly into your html:


<script src="--sdk-directory--/dist/bundles/Carbon.sfx.js"></script>

After that, a variable named Carbon will be accessible in the global scope. This variable will serve as both a class constructor and a namespace. It's usage as a class constructor is the same as when using ES2015 Modules, but its namespace nature is unique to the Global approach.

Carbon variable as a namespace

The Carbon object chains together all the Carbon modules together, so you can access them even if you are using the Global approach. Each time you see in an example something like: 


import * as Errors from "carbonldp/HTTP/Errors";

You can take the portion after the carbonldp/, replace the /s with . and append that to the Carbon namespace to access the same module. For example, accessing that module using the global namespace would be: 


Carbon.HTTP.Errors
Browser - ES2015 Modules

To be able to use ES2015 modules a transpiler like Babel, Traceur or TypeScript needs to be used. This documentation won't cover the details of setting one up, but we recommend using JSPM as a package manager, configured to use TypeScript as a transpiler. That way you can use ES2015 features and bundle your dependencies together!

Each module in Carbon will export at least one named export. Some of the modules will also export a default export containing the "main" export of the module. That export will also be available as the named export Class. For example: 


// This:
import Document from "carbonldp/Document";
let myDocument:Document = new Document();

// Is the same as this:
import * as Document from "carbonldp/Document";
let myDocument:Document.Class = new Document.Class();

We recommend not using default exports, but we provide them in case you are only using the "main" export of a file.

The semantics behind Carbon's modules are based on the usage of the import all statement. They'll likely change in the future, but because modern IDEs lack proper auto-import functionality (WebStorm is getting closer...), importing everything of a module is faster than doing it selectively. Still, you can use selective imports but you'll likely need to rename the import variables like: 


import { Factory as DocumentFactory, Class as Document } from "carbonldp/Document";

For each folder inside the Carbon source, there is a module at the same level with the same name that exports everything contained by the folder. Example: 


import * as Errors from "carbonldp/HTTP/Errors";
import * as Request from "carbonldp/HTTP/Request";

import * as HTTP from "carbonldp/HTTP";

HTTP.Errors === Errors; // true
HTTP.Request === Request; // true

Now that you have installed Carbon successfully you can create your very first app using the UI App. Click here to find out how.

Creating contexts

Now that you have your brand new app (If you don't have your app yet, please create it! (link) ). Let's start with some basic things you can do, like creating documents, saving documents, creating relation links between Documents etc.

Before you can manipulate Carbon objects, you need to create a Carbon platform context. This context will provide a reference to the Carbon platform instance you are working with. (For more about Carbon contexts see: Contexts)

After you create a Carbon platform context, you need an Application context that references the application on which you will be working on. The app context is retrieve from a previously created application

TypeScript JavaScript 

import * as App from "carbonldp/App";
import Carbon from "carbonldp/Carbon";

let carbon:Carbon = new Carbon(); //platform context
let myAppContext:AppContext;

carbon.apps.get( "your-app-slug-here/" ).then( ( appContext:App.Context ) => {
    myAppContext = appContext; //retrieving you app context
}).catch( console.error );
							




let carbon = new Carbon(); //platform context
let myAppContext;

carbon.apps.get( "your-app-slug-here/" ).then( function( appContext ) => {
    myAppContext = appContext; //retrieving you app context
}).catch( console.error );

The apps.get() method returns a Promise that when resolved, will give you the App Context you need.

Retrieving a Document

Now with your App Context you can start working with your App's data. You'll retrieve your root Document, which is the root of all your Document nodes.

TypeScript JavaScript 

//imports
import * Response from "carbonldp/HTTP/Response";
import * Document from "carbonldp/Document";

let carbon:Carbon = new Carbon(); //platform context
let myAppContext:AppContext;
let rootDocument:PersistedDocument.Class;

carbon.apps.get( "your-app-slug-here/" ).then( ( appContext:App.Context ) => {
    myAppContext = appContext; //retrieving you app context

    myAppContext.documents.get( "/" ).then(
        ( [ document, response ]:[ PersistedDocument.Class, Response.Class ] ) => {
            rootDocument = document;
	});

}).catch( console.error );
							





let carbon:Carbon = new Carbon(); //platform context
let myAppContext;
let rootDocument;

carbon.apps.get( "your-app-slug-here/" ).then( function( appContext ) => {
    myAppContext = appContext; //retrieving you app context

    myAppContext.documents.get( "/" ).then(
        function( [document, response] ) => {
            rootDocument = document;
	});

}).catch( console.error );

The documents.get() method accepts a relative URI. That relative URI is going to be resolved to an absolute URI using the context in which we are working on (in this case appContext). To see to what URI it'll resolve you can execute appContext.resolve( "/" ).

Creating a Document

The root Document that you just retrieved can be seen as a the root folder of your whole structure. It is not recommended to modify it. Instead you should create children Documents to hold your data.

You can start by creating Documents in memory, which will need to be persisted later on. To create an in memory document you can use two methods:

  • Document.Factory.create() to create an empty Document, in which properties, fragments and named Fragments could be defined later on.
  • Document.Factory.createFrom() to create a Document from a javascript object.
    • .

In both cases, the document created will not be a persisted Document, until you specifically do so as will be discussed in section Persisting a Document.

In an in memory document you add properties as you will do it in any javascript object. To add fragments or Named Fragments you should use the methods document.createFragment() or document.createNamedFragment() respectively. In the next example, you will see how to create an in memory Document using both methods discussed, as well as how to add properties, fragments and named fragments to a Document.

TypeScript JavaScript 

//imports
import * as Document from "carbonldp/Document";
import * as Fragment from "carbonldp/Fragment";
import * as NamedFragment from "carbonldp/NamedFragment";

//Create an empty Document
let inMemoryDocument:Document.Class = Document.Factory.create();

//Create a Document from an object
let inMemoryOtherDocument:Document.Class = Document.Factory.createFrom( {
    //Document properties
    someProperty: "Property value",
    anotherProperty: 123,
    data: [
        // Fragment
        {
            someProperty: "hello"
        },
        // Named Fragment
        {
            id: "#my-slug",
            someOtherProperty: "good-bye"
        }
    ]
} );

//Add properties, fragments and namedFragments to a Document
inMemoryDocument.someProperty = "Property value";
inMemoryDocument.anotherProperty = 123;

let fragment:Fragment.Class = inMemoryDocument.createFragment( {
    someProperty: "hello"
} );

let namedFragment:NamedFragment.Class = inMemoryDocument.createNamedFragment( "my-slug", {
    someOtherProperty: "good-bye"
} );

Persisting a Document

A persisted Document is a Document that has been saved in Carbon through the method createChild(). In Carbon we handle the fundamental parent-child relation between Documents. Every Document must be the child of a previous existing Document, being the root Document the first parent in your application, as mentioned in Retrieving a document, the root Document id is "/".

The createChild() method can be invoked with different signatures. Is a requisite to provide the parent id and a child Document or javascript object from which the child Document can be created.

TypeScript JavaScript 

//...
import * as Pointer from "carbonldp/Pointer";
import * as Response from "carbonldp/HTTP/Response";

//.. Declare required contexts, create in memory Document

platformContext.apps.getContext( "your-app-slug-here/" ).then( ( appContext:App.Context ) => {
    myAppContext = appContext;

    //Persist the in memory Document: myDocument as a child of "parent-id/"
    myAppContext.documents.createChild( "parent-id/", myDocument ).then(
        ( [ documentPointer, response ]:[ Pointer.Class, Response.Class ] ) => {
            console.log( documentPointer.id); //your persisted Doc id
    });

    //Persisting an object as a child of the root Document
    myAppContext.documents.createChild( "/", {
        name: "Miguel",
        data: [
            // Fragment
            {
                someProperty: "hello"
            },
            // Named Fragment
            {
                id: "#my-slug",
                someOtherProperty: "good-bye"
            }
        ]
    } ).then(( [ documentPointer, response ]:[ Pointer.Class, Response.Class ] ) => {
            console.log( documentPointer.id); //your persisted Doc id
    });

}).catch( console.error );

The documents.createChild() method returns an unresolved Document Pointer, that when resolved allows you access to the new Document's content. You can provide an id to the Document you are creating as a second parameter of the method. The id is known as the child-id/ and it can be any string that will be taken as a suggestion by Carbon and will ultimately be converted into a valid id for the Document you are creating. With this id the new Document can be retrieve, through the method get() or, as mentioned before, through the resolution of the Document Pointer. This will be discussed in the section Resolving a DocumentPointer.

The next sample code shows the method createChild used with the child-id parameter.

TypeScript JavaScript 

//...
import * as Pointer from "carbonldp/Pointer";
import * as Response from "carbonldp/HTTP/Response";

//...

platformContext.apps.getContext( "your-app-slug-here/" ).then( ( appContext:App.Context ) => {
    myAppContext = appContext;

    //Persisting the object as a child of the root Document, with the id child-id
    myAppContext.documents.createChild( "/", "child-id", {
        name: "Miguel",
        data: [
            // Fragment
            {
                someProperty: "hello"
            },
            // Named Fragment
            {
                id: "#my-slug",
                someOtherProperty: "good-bye"
            }
        ]
    });

    //Persist myDocument as a child of "parent-id/", with an id "my new child"
    myAppContext.documents.createChild( "parent-id/", "my new child", myDocument ).then(
        ( [ documentPointer, response ]:[ Pointer.Class, Response.Class ] ) => {
            console.log( documentPointer.id); //the id of you newly created Document
    });

}).catch( console.error );

Resolving a Document Pointer

In the last section you created a persisted Document obtaining an unresolved Document Pointer to your new Document, with the property documentPointer.id. To access the Document information you need to resolve the Document Pointer, as shown in the next example.

TypeScript JavaScript 

//...
import * as PersistedDocument from "carbonldp/PersistedDocument";

//...
let documentPointer:Pointer.Class;
let myPersistedDocument:PersistedDocument.Class;

platformContext.apps.getContext( "your-app-slug-here/" ).then( ( appContext:App.Context ) => {
    myAppContext = appContext;

	myAppContext.documents.createChild( "/", myDocument ).then(
        ( [ documentPointer, response ]:[ Pointer.Class, Response.Class ] ) => {

            documentPointer.resolve().then(
                ( [ document, response ]:[ PersistedDocument.Class, Response.Class ] ) => {
                    myPersistedDocument = document;
			});
		});
}).catch( console.error );
							




//...
let documentPointer;
let myPersistedDocument;

platformContext.apps.getContext( "your-app-slug-here/" ).then( function( appContext ){
    myAppContext = appContext;

	myAppContext.documents.createChild( "/", myDocument ).then(
        function( [ documentPointer, response ] ){

            documentPointer.resolve().then(
                function( [ document, response ] ){
                    myPersistedDocument = document;
			});
		});
}).catch( console.error );

Modifying and Saving a Document

All the Documents in Carbon can be modified. However all the changes done to a Document, are done in memory. To persist the modifications the Documents needs to be saved.

To modify a Document you should:

  • Retrieve the Document:
    • get() using the Document id as describe in Retrieving a Document
    • documentPointer.resolve() resolving a Document Pointer.
  • Modify the Document. Add, modify or delete properties, fragments, named fragments.
  • Save the changes through document.save() method.
TypeScript JavaScript 

//imports

//declarations

platformContext.apps.getAppContext( "your-app-slug/" ).then(
    ( appContext:App.Context ) =>{

        appContext.documents.get( "your-document-id/" ).then(
            ( [ document, response ]:[ Document.Class, Response.Class ] ) => {
                document.newProperty = "You!";
                document.anotherProperty = anotherDocument;
                delete document.someOldProperty;

                document.save().then(
                    ( [ document, response ]:[ PersistedDocument.Class, Response.Class ] ) =>{
                        console.log( document );
                });
        });
}).catch( console.error() );
							



//declarations

platformContext.apps.getAppContext( "your-app-slug/" ).then(
    function( appContext ){

        appContext.documents.get( "your-document-id/" ).then(
            function( [ document, response ] ){
                document.newProperty = "You!";
                document.anotherProperty = anotherDocument;
                delete document.someOldProperty;

                document.save().then(
                    function( [ document, response ] ){
                        console.log( document );
                });
        });
}).catch( console.error() );

Deleting a Document

The method delete of Carbon.Documents class allows you to delete any persisted Document, by providing its URI

TypeScript JavaScript 

import * as App from "carbonldp/App";
import Carbon from "carbonldp/Carbon";

let platformContext:Carbon = new Carbon();

platformContext.apps.getAppContext( "your-app-slug/" ).then( ( appContext:App.Context ) =>{

    appContext.documents.delete( "documentURI" );

}).catch( console.error() );
							




let platformContext = new Carbon();

platformContext.apps.getAppContext( "your-app-slug/" ).then( function( appContext:App.Context ){

    appContext.documents.delete( "documentURI");

}).catch( console.error() );

The delete() methods returns a promise with the server's response.

Linking Properties

Carbon Documents can be related to other Documents through their properties. Properties to relate any two given Documents can be defined manually. However, to simplify the relation management Carbon handles specific properties such as contains and members, belonging to the parent-child relation and membership relation respectively. Carbon provides you with methods to handle both, this properties and the list of Documents related through them.

Parent-Child relation

The parent child relation exists as a rule for all Documents and is automatically maintained by Carbon as a hard link between two Documents. When a new Document is persisted, the parent-child relation is created and from this you can obtained all the Documents created as children of any given Document, in the form of pointers or a list. This can be accomplished through the methods:

  • listChildren: Return all the children of the container specified.
  • getChildren: Retrieves all the children of a document and their content.

From Carbon.Documents class.

TypeScript JavaScript 

//imports

//declarations

platformContext.apps.getAppContext( "your-app-slug/" ).then(
    ( appContext:App.Context ) => {
        myAppContext = appContext;

        myAppContext.documents.getChildren( "parentURI" ).then(
        ( [ children, response ] : [ PersistedDocument.Class[], HTTP.Response.Class] ) => {
            console.log( children );
        });

        myAppContext.documents.listChildren( "parentURI" ).then(
        ( [ Children, response ] : [ Pointer.Class[], HTTP.Response.Class]) => {
            console.log( "list of children:", children );
        });
});
							



//declarations

platformContext.apps.getAppContext( "your-app-slug/" ).then(
    function( appContext ){
        myAppContext = appContext;

        myAppContext.documents.getChildren( "parentURI" ).then(
        function( [ children, response ] ){
            console.log( children );
        });

        myAppContext.documents.listChildren( "parentURI" ).then(
        function( [ Children, response ] ){
            console.log( "list of children:", children );
        });
});

Membership relation

The membership relation is a property that exists in all Carbon Documents, that allows to relate your Document with another Document through a soft link. All Carbon Documents can manage one member list per Document. Members can be added to, removed from, listed or retrieved using this methods.

  • addMember | addMembers: Add the specified resource/resources URIs or Pointers as member/members of the document container specified.
  • removeMember | removeMembers: Remove the specified resource/resources URIs or Pointers as member/members of the document container specified.
  • removeAllMembers: Remove all the members of the document container specified.
  • listMembers: Retrieves all the members of a document with out resolving them.
  • getMembers: Retrieves all the members of a document and their contents.

From Carbon.Documents class.

TypeScript JavaScript 

//imports

//declarations

platformContext.apps.getAppContext( "your-app-slug/" ).then(
    ( appContext:App.Context ) =>{
        myAppContext = appContext;

        myAppContext.documents.listMembers( "parentURI/" ).then(
            ( [members, response]:[Pointer.Class[], Response.Class] ) =>{
                let myPromises:Promise<Response.Class>[] = [];

                console.log( "members: ", members );
                    //members: member4, member1, member3, member2

                myPromises.push( myAppContext.documents.addMember("parentURI/", "memberToAddURI/"));
                myPromises.push( myAppContext.documents.removeMember("parentURI/", "member2/"));

                return Promise.all( myPromises );

        }).then( ( responses:Response.Class[] ) => {
            myAppContext.documents.listMembers( "parentURI" ).then(
                ( [ members, response ] : [ Pointer.Class[], Response.Class ] ) =>{

                    console.log( "members: ", members );
                        // members: member1, member4, memberToAddURI, member3
        });
});
								



//declarations

platformContext.apps.getAppContext( "your-app-slug/" ).then(
    function( appContext ){

Creating an Access Point

As mentioned in the previous section, Documents can handle up to one member relation within them. To manage several member relations Documents need the help of Access Points. (link to Access Points object model)

First you need to create the Access Point in Memory. Like a Document, an Access Point can be created as a new empty Access Point or from a pre existing javascript object with any number of properties, using the methods:

  • create() creates an Access Point
  • createFrom() creates an Access Point from a javascript object

From the Carbon.AccessPoint.Factory class. Both of this methods have two required parameters and an optional one:

  • membershipResource
  • hasMemberRelation
  • memberOfRelation (optional)

TypeScript JavaScript 

//imports

//declarations
let myObject:Object  = { name: "myNieces", description: "All the nieces of myParent."};
let myAccessPoint:AccessPoint.Class;
let nephews:AccessPoint.Class;
let nieces:AccessPoint.Class;

//create() create an in memory access point
myAccess = AccessPoint.Factory.create( myParent, "documentRelation", "inverseDocumentRelation" );
nephews = AccessPoint.Factory.create( myParent, "nephew", "uncle" );

//createFrom() create an in memory access point from an object.
nieces = AccessPoint.Factory.createFrom( myObject, myParent, "niece", "aunt" );
							

//imports

//declarations
let myObject = { name: "myNieces", description: "All the nieces of myParent."};
let myAccess;
let nephews;
let nieces;

//create() create an in memory access point
myAccess = AccessPoint.Factory.create( myParent, "documentRelation", "inverseDocumentRelation" );
nephews = AccessPoint.Factory.create( myParent, "nephew", "uncle" );

//createFrom() create an in memory access point from an object.
nieces = AccessPoint.Factory.createFrom( myObject, myParent, "niece", "aunt" );

Both methods return a Pointer to the Access Point in memory.

Persisting an Access Point

After creating an Access Point in memory it must be persisted.

Now that you have an Access Point object you can create a Persisted Access Point, by passing an in memory Access Point to the createAccessPoint() method from Carbon.Documents class. This methods will create a persisted Access Point for a document specified in the "parent" URI passed as a parameter. Remember this URI must match the URI of the membership Resource defined in the Access Point in memory. You can also call this method from a Persisted Document with the same name method, createAccessPoint() from Carbon.PersistedDocument.Factory class.

TypeScript JavaScript 

//imports

let myPersistedDocument:PersistedDocument.Class;
let myParent:Pointer.Class;
let myAccessPoint:AccessPoint.Class;

//create() access point in memory
myAccessPoint = AccessPoint.Factory.create( myParent, "documentRelation", "inverseDocumentRelation" );

//createAccessPoint() creates a persisted access point from an in memory access point
AppContext.documents.createAccessPoint( "parentURI" , myAccessPoint );

//createAccessPoint() creates a persisted access point for a persisted document
myPersistedDocument.createAccessPoint( myAccessPointObject );
							

//imports

let myPersistedDocument;
let myParent;
let myAccessPoint;

//create() access point in memory
myAccessPoint = AccessPoint.Factory.create( myParent, "documentRelation", "inverseDocumentRelation" );

//createAccessPoint() creates a persisted access point from an in memory access point
