What is a context?

A Context is a reference to the place where you will be working on the server, by declaring a context you are pointing to where your information should go. When you use Carbon modules and methods, they use the definitions in your context to channel the information correctly. Without a context, every time you wish to write something in your application you would have to specify the whole address of your target, as well as a bunch of information you need to make requests to the server.

When using a context you can define settings, authentication services and the object schema, once at the beginning of your application, allowing you to set your domain, your authentication information, etc. By configuring this information in your context, you will be able to simplify the programming process e.g. setting your domain in your context allows you to use relatives URIs through out your code, the relative URIs will be resolved against your contexts, making your code more or less independent from the location of your platform.

There are two types of contexts that you can work with:

  • Platform context
  • App Context

Both types of contexts belong to different classes that extend AbstractContext Class, which in turn extends SDKContext Class, that will be discussed in the next sections.

Platform context

All contexts reference, as we said before, the destination in our server where our information will go. The platform context references your entire Carbon platform. It contains the domain of where your platform is, and it gives you access to define/modify settings, authentication services and object schemas, that will be available globally for the Carbon SDK Modules to use. The platform context belongs to Carbon Class

Declare the platform context

To declare a platform context you need to import Carbon from the carbonldp/Carbon library and instance a new Carbon object, this will be your platform context. You can get the default base uri value against which relative URIs will be resolved in the platform context by resolving the relative uri. e.g. the default platform context base URI is https://carbonldp.com/platform/

TypeScript JavaScript 

import Carbon from "carbonldp/Carbon";

let carbonContext:Carbon = new Carbon();

carbonContext.resolve( "" ); // https://carbonldp.com/platform/


								

import Carbon from "carbonldp/Carbon";

let carbonContext = new Carbon();

carbonContext.resolve( "" ); //https://carbonldp.com/platform/

The context.resolve() method resolve the relative URI provided( "" ) to an absolute URI (https://dev.carbonldp.com/platform/) using the context in which you are working on (in this case carbonContext).

Access/Modify your platform context

Now that you have declared a platform context, you can work with it.

In the next code sample, you can see a platform context that has been set with a different domain than the default value which is https://carbonldp.com/ . This means that all the relative URIs used in your applications will be resolved against this new domain.

Following this pattern you will be able to use all the methods and access all the properties of your carbon context, to see the Carbon class specification click here

TypeScript JavaScript 

import Carbon from "carbonldp/Carbon";

let carbonContext:Carbon = new Carbon();

carbonContext.setSetting( "domain", "dev.carbonldp.com" );

carbonContext.resolve( "" ); // https://dev.carbonldp.com/platform/

								

import Carbon from "carbonldp/Carbon";

let carbonContext = new Carbon();

carbonContext.setSetting( "domain", "dev.carbonldp.com" );

carbonContext.resolve( "" ); //https://dev.carbonldp.com/platform;

A platform context default settings are:

  • domain: "carbonldp.com"
  • http.ssl: true
  • auth.method: Carbon.Auth.Method.TOKEN
  • platform.container: "platform/"
  • platform.apps.container: "apps/"
  • platform.agents.container: "agents/"
  • vocabulary: "vocabulary/#"

You can add as many personalized settings as you like.

Application context

An application context references an application that already exists in your platform context. and it belongs to the Carbon.App.Context Class.

The application context allows you to set specific settings and information that will be shared and used by all the Carbon modules and methods. You can configure the roles and agents specific to the application, and you can see the information contained in it. Through the application context you can access the same settings, authentication services or object schemas as from your platform context. To see more in depth information about the App Context Class go to the Carbon.App.Context technical specifications

Getting the App Context

Once you have your platform context declared, you can get the application context that will reference it. With this context you can use all the modules and methods available to the application, for example: to retrieve the Documents that live inside your application, or to create new information.

In the code below you can see how to retrieve a context of your application named "test-app". To have an application context, you need to have had created the application first, if you do not have an application yet, please create it

TypeScript JavaScript 

import Carbon from "carbonldp/Carbon";

let carbonContext:Carbon = new Carbon();

carbonContext.apps.getAppContext( "test-app/" ).then(
	( appContext:App.Context ) => {
		appContext.resolve( "" ); // https://carbonldp.com/apps/test-app/
	}
);

								

import Carbon from "carbonldp/Carbon";

let carbonContext = new Carbon();

carbonContext.apps.getAppContext( "test-app/" ).then(
	(appContext) =>{
		appContext.resolve( "" ); //https://carbonldp.com/apps/test-app/
	}
);

The apps.get() method returns a Promise that when resolved, gives you the App Context you need.

Inheritance

In this section we will discuss the inheritance of settings and authentication services, the object schema inheritance will be discussed in the Object Schema Documentation, as is a more complex subject.

The application context has access to all the settings, services and object schema contained in the platform context in which it lives, unless those settings are explicitly overwritten since they are not defined by default. This means that when there is an attempt to access an application context's setting, it is first searched for in the application context, and if not defined there, then it is searched for in the platform context. If it is not found in either context, then this means the setting has not been defined at all. This type of inheritance is similar to the prototypal inheritance. Bellow you will find an example of settings' inheritance.

For this next example assume you have two applications: test-app and your-app, from which you wish to set and get the setting "name" in both, your platform and application context. These applications live in carbonldp.com, and remember that to get and set the settings you need to instance a platform context, and the application contexts for both apps.

TypeScript JavaScript 

import Carbon from "carbonldp/Carbon";
import * as App from "carbonldp/App";

let carbonContext:Carbon = new Carbon();
let testAppContext:App.Context;
let yourAppContext:App.Context;

//Get your application contexts
carbonContext.apps.getAppContext( "test-app/" ).then(
	( appContext:App.Context ) => {
		testAppContext = appContext;
	}
);
carbonContext.apps.getAppContext( "your-app/" ).then(
	( appContext:App.Context ) => {
		yourAppContext = appContext;
	}
);

//Default Value of name Setting =  null,
//since the name setting has not been set and it is not predefined in the Platform Context settings
carbonContext.getSetting( "name" );  //null
testAppContext.getSetting( "name" ); //null
yourAppContext.getSetting( "name" ); //null


//Set name setting in test-app app context
testAppContext.setSetting( "name", "TestApp" );

carbonContext.getSetting( "name" );  //null
testAppContext.getSetting( "name" ); //TestApp
yourAppContext.getSetting( "name" ); //null


//Set name setting in the platform context
carbonContext.setSetting( "name", "CarbonPlatform" );

carbonContext.getSetting( "name" );  //CarbonPlatform
testAppContext.getSetting( "name" ); //TestApp
yourAppContext.getSetting( "name" ); //CarbonPlatform

//Deleting an app setting
testAppContext.deleteSetting( "name" );

carbonContext.getSetting( "name" );  //CarbonPlatform
testAppContext.getSetting( "name" ); //CarbonPlatform
yourAppContext.getSetting( "name" ); //CarbonPlatform
